## Что было сделано:

### 1. Замена APScheduler на lifespan + asyncio

**Проблема:** APScheduler не был установлен и вызывал падение сервера.

**Решение:** Заменили устаревший механизм `@app.on_event("startup")` на современный `lifespan` context manager с использованием встроенного `asyncio`.

**Изменения в `backend/app/main.py`:**
- Добавлен `@asynccontextmanager` для управления жизненным циклом приложения
- Создана асинхронная функция `update_news_async()` для обновления новостей
- Создана фоновая задача `news_update_background_task()` с бесконечным циклом
- Синхронный парсер новостей обернут в `asyncio.to_thread()` для неблокирующего выполнения
- Удалены устаревшие `@app.on_event("startup")` и `@app.on_event("shutdown")`

**Преимущества:**
- Не требует установки дополнительных библиотек
- Встроенный механизм FastAPI
- Асинхронный и не блокирует сервер
- Автоматически останавливается при shutdown
- Первое обновление через 30 секунд после запуска, затем каждые 6 часов

## Технические детали:

### Механизм lifespan:
```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup
    task = asyncio.create_task(news_update_background_task())
    yield  # Приложение работает
    # Shutdown
    task.cancel()
```

### Фоновая задача:
```python
async def news_update_background_task():
    await asyncio.sleep(30)  # Ждем запуска сервера
    await update_news_async()  # Первое обновление
    while True:
        await asyncio.sleep(6 * 60 * 60)  # 6 часов
        await update_news_async()
```

### Обертка синхронного кода:
```python
news_list = await asyncio.to_thread(fetch_news_from_website)
await asyncio.to_thread(save_news_to_db, news_list, db)
```
